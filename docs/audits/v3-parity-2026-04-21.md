## Agent 1 — prompt system + dev workflow

**Verdict: BETTER_THAN_V3.** v4's SoT discipline, hot-reload on transitive fragment change, fragment reuse, per-agent fingerprint map, and registry tests all exceed v3.

**Positives confirmed:**
- Both have a centralized registry loader; v4's is slightly stricter (no inline fallback, unlike v3's `register_inline()`).
- v4 rechecks every dependency's mtime per call in dev (`src/lib/prompts/registry.ts:141-167`); v3's equivalent only rereads the named file, no transitive fragment detection.
- v4 genuinely USES fragments — 14 agent prompts + KA Block 1 pull `structured_output_contract`, Block 1 pulls `style_opus_voice`. v3 has the composition machinery but doesn't use it (no `get_composed()` calls in `src/agents/` or `src/core/` outside docstrings).
- v4 persists `{agentName → fingerprint}` map per turn; v3 persists a scalar for KA only.
- v4 has `pnpm prompts:dump`; v3 has no equivalent.
- v4 has `src/lib/prompts/__tests__/registry.test.ts` (9 cases); v3 has none.

**Regressions / gaps vs v3:**
- [MINOR] No prompt-eval harness analogous to v3's `src/prompts/eval.py` (fixture-based regression harness for prompt unit testing).
- [MINOR] No prompt dependency-graph tool like v3's `PromptDependencyGraph` (`depends_on` frontmatter + mermaid export + `impact_report()` — "what downstream prompts does editing this fragment affect?").
- [MINOR] Unfilled `{{var}}` placeholders fail silently in v4; v3 logs them as debug.

## Agent 2 — KA narrative craft machinery

**Verdict: MIXED.** Most v3 craft machinery is present as real runtime code (not just prompt scaffolding) — sakuga ladder, style-drift shuffle bag, vocab-freshness regex, portrait extractor, composition-mode selector all exist as working TypeScript. But one BLOCKER hides behind this: the workflow never calls the composition-mode selector, and two MAJOR call-site gaps defeat the vocab-freshness check.

**Positives confirmed via my personal reads:**
- `src/lib/ka/sakuga.ts:27-34` — SAKUGA_PRIORITY ladder ported verbatim (6 triggers, correct mode mapping). Fallback path for CLIMACTIC weight. Wired into `turn.ts:464` → Block 4 `{{sakuga_injection}}`. Unit tests lock the order.
- `src/lib/ka/diversity.ts:27-36` — STYLE_DRIFT_POOL has all 8 directives verbatim. `pickStyleDrift` implements convergence check (last 3 openings, Set-size ≥ 2 skips) + intent disallow + CLIMACTIC disallow. Wired into Block 4.
- `src/lib/ka/diversity.ts:124-135` — 7 regex patterns for simile/personification/negation triples. `detectStaleConstructions` implements threshold (3 default), proper-noun skip, jargon filter.
- `src/lib/ka/portraits.ts:29-46` — `extractNames()` scans `**Name**` post-KA. Passive per v3. KA's prompt at `fragments/style_opus_voice.md:25` teaches bolding without mentioning portraits.

**BLOCKER:**
- [BLOCKER] **Effective composition / mode-shift not computed per turn.** `turn.ts:452` hardcodes `compositionMode: "standard"`. `scale-selector-agent.ts` is registered as a KA consultant at `key-animator.ts:87-93` but the workflow NEVER invokes it deterministically. The only path to mode-shift is if KA spontaneously decides to spawn the consultant mid-stream. OJ also receives `"standard"` hardcoded, so it doesn't know about tier-differential reframing. Consequence: a Tier 3 protagonist fighting a Tier 9 mook narrates the same way as Tier 8 vs Tier 8. The `op_dominant` reframe onto meaning/cost instead of survival — the most load-bearing thing v3's composition system exists for — is silently inert.

**MAJOR:**
- [MAJOR] **Vocab freshness whitelists never built at call site.** `turn.ts:489` passes `{ recentNarrations }` only. `properNouns` + `jargonAllowlist` inputs are never populated. The profile has `power_system.limitations`, `author_voice.sentence_patterns`, registered NPC names, but nothing extracts them into the Sets the detector expects. Consequence: false positives hit KA repeatedly — "you've used the name 'Spike' 5 times, vary your prose" despite Spike being the protagonist.
- [MAJOR] **No research phase with intent-adaptive strategies + tight summary + round cap.** v3's two-phase "fast-model research → creative-model write" pattern (key_animator.py:564-718, max_tool_rounds=3, TIGHT summary with RELEVANT FACTS / NPC INTELLIGENCE / CONTINUITY / TACTICAL NOTE sections) is collapsed into "creative model decides and calls tools mid-stream" in v4. This is by design per the KA-as-orchestrator inversion, but no eval gates the regression. KA can churn through tool calls mid-generation without bound.

**MINORS:**
- [MINOR] Style-drift `recentlyUsed` hardcoded empty at `turn.ts:487` — the same directive can fire twice in a row.
- [MINOR] Power-system `limitations` is JSON-dumped via `JSON.stringify(profile.ip_mechanics.power_system)` at `blocks.ts:258`. v3 explicitly formatted `**LIMITATIONS (You MUST Respect These):**` with directive weight. v4 loses the emphasis.
- [MINOR] `turn.ts:587` hardcodes `portraitMap[n] = null` — `buildPortraitMap` with a real resolver is never invoked. Acceptable per known pending ProductionAgent; but the machinery is half-built.
- [MINOR] v3's Approach D (BEAT_CRAFT_GUIDANCE — 7 arc-phase × writing-craft directive map) is fully absent from v4. Block 4's `arc_phase` renders as a value with no craft directive attached.

## Agent 3 — memory system architecture

**Verdict: MAJOR STRUCTURAL GAPS.** Of the 7 subsystems v3 has, v4 has 1 genuinely live (episodic tsvector search via `recall_scene`), 4 scaffolded-but-inert (categories, retrieval tiers, heat column, ranker agent), and 2 MISSING ENTIRELY (context blocks, rule library).

**BLOCKERs:**
- [BLOCKER] **Context blocks — MISSING ENTIRELY.** v3 has `context_blocks` table + `ContextBlockStore` + `ContextBlockGenerator` — per-entity living prose summaries (arc/thread/quest/npc/faction) with continuity checklists, versioned, status-tracked, injected at session start. Zero hits on `context_block | ContextBlock | arc_block | quest_block | npc_block | faction_block` anywhere in v4's `src/`. v4 has the structured data (NPCs as rows, arcs as history) but no distilled living summary. KA forced to rebuild NPC+arc+quest understanding from scattered MCP tool calls every session.
- [BLOCKER] **Rule library — MISSING ENTIRELY.** v3 has static YAML (13 files: scales, power_tiers, dna_scales, op_expressions/focuses/presets/tensions, compatibility_matrix, ceremonies, archetypes_legacy, genres + subdir) indexed into `rule_library_chunks` table with pgvector + specialized getters (`get_dna_guidance(axis, value)`, `get_power_tier_guidance`, etc.). v4 has a single placeholder — `blocks.ts:269` renders `session_rule_library_guidance` from `input.sessionRuleLibrary` with fallback `"(no rule-library guidance loaded this session)"`. `turn.ts` does not pass this field at all. No table, no YAML, no getters. Block 1 dumps `heroism: 7` as a number with no "what 7 means." 24 DNA axes + 13 composition axes become form fields instead of prescriptive pressures.
- [BLOCKER] **Semantic retrieval runtime — empty.** `search_memory` tool returns `{memories: []}` (`semantic/search-memory.ts:59-61`). No embed call, no cosine query, `embedding` column is jsonb null default. MemoryRanker agent is wired but has no upstream retrieval to rerank. Schema ready + ranker ready + zero execution. Chronicler writes `semantic_memories` rows; nobody reads them. The one live retrieval path is `recall_scene` (plainto_tsquery + ts_rank against `narrative_tsv`) — that's the episodic layer only.

**MAJORs:**
- [MAJOR] **Heat decay + boost + floor physics — 4 of 4 missing.** Default heat is 50 (`schema.ts:309`) not 100 (v3). No decay code anywhere. No boost-on-access code. No floor logic. No `flags jsonb` column for `plot_critical | milestone_relationship` markers. Every memory sits at insert-time heat forever. Also: `get_critical_memories` stubbed to return `{memories: []}`, no way to mark critical even if decay landed.
- [MAJOR] **Retrieval tier drift.** v4's `retrievalBudget(epicness)` uses 0.25/0.5/0.75 thresholds (turn.ts:72-77) where v3 uses 0.3/0.6. COMBAT floor (Tier 2 minimum for combat) is absent. `special_conditions` bump (+1 tier when special_conditions present) is absent. `is_trivial_action` gate (epicness<0.2 AND intent not in {COMBAT,ABILITY,SOCIAL} AND no special_conditions → 0 memories) is absent.
- [MAJOR] **Multi-query decomposition.** v3's `_decompose_queries` builds 2-3 queries (action/situation/entity-focused), `_multi_query_search` merges with dedup. v4's `search_memory` input schema takes a single `query` string; no interface for KA to fan out; no server-side dedup. MINOR now (retrieval is empty anyway), MAJOR once retrieval lands.

**MINORS:**
- [MINOR] Static-boost step (+0.3 for session_zero/plot_critical, +0.15 for episode) isn't mentioned anywhere in v4. Defers until retrieval lands.

## Agent 4 — dialectic + authority gradient

**Verdict: CRITICAL WRITE-PATH GAPS.** The routing side of the three-authority-source model is complete, but the write paths that make each authority kind actually BIND are mostly stubs. Override classification happens; persistence doesn't. WB entity acceptance happens; catalog writes don't fire. Meta conversation loop is completely absent.

**BLOCKERs:**
- [BLOCKER] **Override persistence never fires.** Router classifies `/override` into one of 4 categories, workflow short-circuit at `turn.ts:369-405` writes the turn row but NEVER appends to `campaign.settings.overrides`. Next turn reads `priorOverrides` empty. The router's own comment (`router.ts:28`) claims "the workflow step consumes the verdict and writes" — it doesn't. Compare v3 `override_handler.py:100-112` which commits to DB. Consequence: `/override Jet cannot die` gets an ACK, then is forgotten.
- [BLOCKER] **WB-accepted entity persistence evaporates.** `WorldBuilderOutput.entityUpdates` (`world-builder.ts:50-54`) is populated by the prompt on ACCEPT; the workflow's worldbuilder-branch reads it nowhere. Compounding: WB short-circuits BEFORE KA runs, so KA doesn't see the assertion either. Player says "Jet is actually a retired ISSP major" → in-character ACK → next turn, gone. v3 ran KA after WB, so KA could `register_catalog_npc` itself.

**MAJORs:**
- [MAJOR] **Meta conversation loop entirely absent.** Zero hits on `meta_conversation | /resume | /play | /back | /exit` in v4's `src/`. No `_in_meta_conversation` state, no multi-turn Director+KA dialectic, no `/resume {suffix}` pass-through. v4's route handler at `turn.ts:382` assigns `turnNumber: ctx.nextTurnNumber` and increments, so `/meta` consumes a game turn. v3's authorship-calibration conductor loop is gone with no replacement documented in vision files.
- [MAJOR] **Transient vs catalog NPC distinction dropped.** v3 has two tools: `spawn_transient(name, description)` for scene-local flavor (no portrait, no persistent catalog row); `register_catalog_npc(name, role, visual_tags, personality)` for permanent NPCs (triggers portrait + relationship tracking). v4 has only `register_npc` in `src/lib/tools/chronicler/register-npc.ts`. Chronicler's prompt (`chronicler.md:11, 19-23`) explicitly biases "when in doubt, catalog" — opposite of v3. Every named bartender, guard, or passerby gets a catalog row. v3's catalog-inflation failure mode is reopened by design.
- [MAJOR] **WB reshape to editor-not-gatekeeper not landed.** `memory/project_worldbuilder_as_editor.md` is locked (2026-04) with the redesign: drop REJECT, default ACCEPT, CLARIFY only for local physical ambiguity, add non-blocking FLAG for craft concerns. `world-builder.ts:28-33` code comment acknowledges the reshape is pending. Prompt still teaches REJECT + canon-contradiction rejection — the reshape is documented but unimplemented.
- [MAJOR] **WB `EntityUpdate` schema thinner than v3 NPCDetails.** `world-builder.ts:50-54` has `{kind, name, details: string}`. v3 has full structured NPCDetails (personality, goals, secrets, visual_tags, faction, knowledge_topics, power_tier, ensemble_archetype). When persistence lands, Chronicler will have to re-parse prose instead of consuming structured fields.

**MINOR:**
- [MINOR] **KA loses override category tag in Block 4.** `turn.ts:547` passes `.value` strings only, dropping the `[CATEGORY]` prefix v3 used. NPC_PROTECTION vs TONE_REQUIREMENT semantic hint lost.

### Intentional v4 inversions (defensible, NOT gaps)

- **KA orchestrates instead of pipeline calling KA last.** Architectural lock-in per `project_v4_vision.md`. Research-phase collapse is a consequence; MAJOR #12 defends this design but flags the testing gap.
- **Memory fans out from one retrieval into seven cognitive MCP surfaces.** Different retrieval shape per layer.
- **WB editor-not-gatekeeper.** Locked per `project_worldbuilder_as_editor.md`. Reshape not yet implemented, but direction is canonical. MAJOR #13 is "land the locked design," not "reverse it."
- **Three-layer DNA/composition model** (canonical → active → arc_override). v4 addition.
- **24 DNA axes** vs 11; **13 composition axes** vs 3. Expanded drift surface.
- **SZ one conductor conversation** vs four-stage pipeline. Not directly audited.
- **Chronicler as unified orchestrator** vs five parallel post-turn workers.

### Genuine v4 improvements over v3

- Fragment composition actually used (v3 had machinery but didn't use it).
- Prompt fingerprint per-agent map vs scalar.
- Hot-reload on fragment change transitively.
- `pnpm prompts:dump` (no v3 equivalent).
- Registry tests (no v3 equivalent).
- Per-campaign provider + tier_models (v4) vs single-provider (v3).
