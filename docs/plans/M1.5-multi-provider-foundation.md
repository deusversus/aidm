# M1.5 — Multi-provider foundation

**Status:** Drafted 2026-04-20 after void-mode strategic conversation.
**Goal:** per-campaign provider + tier_models architecture shipped with Anthropic populated. Google / OpenAI / OpenRouter slots stubbed in the registry so later milestones (M3.5 Google-KA, M5.5 OpenAI-KA + OpenRouter shim) plug in without architectural rework.

---

## The orienting principle

**Build the whole shape.** Same inversion as M1: the architecture at M1.5 is the architecture at M6.5, with more content inside the slots. Four providers × four tiers × per-campaign config is the scaffold. Anthropic populated today; Google / OpenAI / OpenRouter populated as their KAs land.

A provider entry that returns an empty roster is a live slot with no models yet — not a missing feature. Users who try to create a campaign on Google today see "Google-KA lands in M3.5; currently available: Anthropic" — a helpful error, not a crash.

---

## Definition of done

- `src/lib/providers/` exists and is the source of truth for what models are selectable per provider per tier.
- Campaigns carry `provider` + `tier_models` in `settings`; validated against the provider's registry entry on write.
- Every hot-path LLM call (router pre-pass, consultants, KA) resolves `{ provider, model }` from the campaign context passed into it — no global env read in the hot path.
- `src/lib/env.ts` `tiers` table renamed to `anthropicDefaults`; purpose narrowed to fallback + new-campaign-seed.
- Settings UI lets the user pick provider + per-tier model on a campaign; only Anthropic's roster is populated right now.
- Existing Bebop campaign plays unchanged after migration (backfilled to Anthropic defaults).
- New campaign created with a non-default creative model (e.g., Sonnet 4.6 or an older Opus snapshot) plays to completion and the voice-shift is observable.
- 174+ existing tests continue to pass; new tests cover the provider registry + schema validation + dispatch.

**What does NOT land in M1.5:**

- Google-KA implementation. Google slot exists in registry, populated with models from the 2026-04 list, but the KA code is stub/throw-until-M3.5.
- OpenAI-KA / OpenRouter shim. Same treatment.
- Billing substrate. M2.5.
- Multi-provider failover. M6.5.
- Feature-flag / compat layer. M6.5.
- Per-user "default provider for new campaigns" pref. Nice-to-have; defer to M3 or later.

---

## Risks carried in

1. **Runner-refit scope creep.** The `_runner.ts` + `IntentClassifier` refit touches ~15 call sites. Each needs `{ provider, model }` plumbed through. Mechanical, but easy to miss a site; audit catches this.
2. **Campaign migration.** Today there's one Bebop campaign on one user. Backfill is trivial now; it stops being trivial later. Ship the migration before any user beyond jcettison creates a campaign.
3. **UI state management.** Provider dropdown → tier-model dropdowns repopulating on provider change is React state that's easy to get wrong (stale values persisting, orphan model IDs, validation timing). Worth a settle-down test pass before audit.
4. **env.ts rename backward compatibility.** Scripts, tests, or deploy paths that import `tiers` by name will break on the rename. Grep before the rename, rewrite all sites in the same commit.

## Commit plan

Each commit follows the audit cadence: **implement → subagent audit → fix findings → push.** Thorough commits, not tiny ones.

### Commit A — Provider registry

`src/lib/providers/` with the scaffolding for all four providers. Anthropic populated. Google / OpenAI / OpenRouter stubbed.

**Deliverables:**
- `src/lib/providers/registry.ts` — exports a map `{ anthropic, google, openai, openrouter }`; each entry is a `ProviderDefinition` with shape:
  ```ts
  {
    id: "anthropic" | "google" | "openai" | "openrouter",
    displayName: string,
    tiers: {
      probe: { defaultModel: string, selectableModels: string[] },
      fast: { defaultModel: string, selectableModels: string[] },
      thinking: { defaultModel: string, selectableModels: string[] },
      creative: { defaultModel: string, selectableModels: string[] },
    },
    features: {
      nativeMCP: boolean,
      promptCaching: "breakpoint" | "context-id" | "system-auto" | "none",
      thinking: "adaptive" | "native" | "reasoning-tokens" | "none",
    },
    available: boolean,  // false until the provider's KA lands
  }
  ```
- `src/lib/providers/anthropic.ts` — populated with the full Anthropic roster (from user-confirmed shortlist): Opus 4.7, Sonnet 4.6, Haiku 4.5 latest + snapshots 4.6 / 4.5 / 4.1 / 4.0 / claude-3-haiku. Probe = Haiku 4.5 only. Features: nativeMCP: true, promptCaching: "breakpoint", thinking: "adaptive".
- `src/lib/providers/google.ts` — populated with the three Gemini IDs user confirmed; `available: false` with reason "Google-KA lands M3.5."
- `src/lib/providers/openai.ts` — empty roster; `available: false` with reason "OpenAI-KA lands M5.5."
- `src/lib/providers/openrouter.ts` — empty roster (free-form field when we populate later); `available: false`.
- Validation helper `validateCampaignProviderConfig({ provider, tier_models })` — asserts provider is known, all tier_models are in the provider's selectable list, surfaces actionable errors.
- Tests: registry shape, validation success/failure cases, Anthropic entry integrity.

**Audit focus:** registry shape is extensible for future providers without refactor; validation rejects cross-provider tier_models; Anthropic entry matches the user-confirmed shortlist exactly.

### Commit B — Campaign schema extension

Zod `Campaign` grows `provider` + `tier_models`. Drizzle migration. Backfill for existing campaigns.

**Deliverables:**
- `src/lib/types/campaign.ts` — Zod schema adds `provider: ProviderId`, `tier_models: TierModels` (under `settings` to preserve the evolve-without-migration property, same as world_state and overrides).
- Migration: zero-schema-change (settings is already jsonb); a one-time DATA migration script (`scripts/migrate-campaign-providers.ts`) that reads all campaigns where settings lacks provider / tier_models and backfills to Anthropic + current `anthropicDefaults` values.
- Seed (`seedBebopCampaign`) writes provider + tier_models on create so new signups get a well-formed campaign.
- Tests: schema round-trip, migration script idempotency, seed output contains the new fields.

**Audit focus:** migration is safe to run N times; no campaign ends up with a null / invalid provider; Zod rejects campaigns with malformed tier_models.

### Commit C — Runner refit

`_runner.ts` + `classifyIntent` + every agent that reads `tiers` globally is refactored to take `{ provider, model }` per call.

**Deliverables:**
- `AgentRunnerConfig` gains `provider: ProviderId, model: string` as required fields per invocation (today it has `tier: "fast" | "thinking"` reading global defaults).
- `classifyIntent` — same refit on the direct-Gemini path.
- Every consultant entry point (`judgeOutcome`, `validateOutcome`, `advisePacing`, `rankMemories`, `produceRecap`, `resolveCombat`, `selectScale`, `handleOverride`, `validateAssertion`) accepts a `modelContext: { provider, tier_models }` arg; internally resolves the right tier's model and passes it into the runner.
- Callers without campaign context (scripts, `/api/ready`, tests) build a synthetic context from `anthropicDefaults`.
- Tests updated to pass model context explicitly; add a test that a call with no context falls back to anthropicDefaults.

**Audit focus:** no call site reads `tiers` globally in the hot path; fallback-to-defaults path only fires for the enumerated exceptions (scripts, health check, tests); every consultant's tests still cover its failure modes.

### Commit D — Turn-pipeline threading

Campaign's `{ provider, tier_models }` flows from turn.ts through router → consultants → KA.

**Deliverables:**
- `TurnWorkflowInput` / `RouterInput` gain model context from the campaign row load.
- Turn workflow builds a `modelContext` once per turn from the loaded campaign and passes it into every agent call it makes.
- KA's `KeyAnimatorInput` gains `modelContext`; `key-animator.ts` reads `modelContext.tier_models.creative` instead of `tiers.creative.model`.
- Subagent `AgentDefinition`s inside KA use `modelContext.tier_models.{fast,thinking}` for their `model` field.
- Integration test: run `runTurn` with a mock campaign configured for Sonnet creative; verify KA is invoked with `model: "claude-sonnet-4-6"` and not Opus.

**Audit focus:** every agent call in the turn loop receives model context; no agent reads global tier config; Sonnet-configured campaign produces KA with Sonnet model string; integration test proves end-to-end.

### Commit E — env.ts rename + narrow

`tiers` → `anthropicDefaults`. Same data, renamed, narrowed purpose.

**Deliverables:**
- `src/lib/env.ts` — `export const anthropicDefaults = { probe, fast, thinking, creative }` replaces `export const tiers = ...`.
- Updated docstring: fallback-only; source of truth is per-campaign tier_models.
- Grep all imports of `tiers`; rename them.
- Imports inside `src/lib/providers/anthropic.ts` reference the same defaults to avoid duplication.
- Commit message highlights the rename is the point — clarity prevents drift.

**Audit focus:** no remaining reference to `tiers` outside anthropicDefaults / providers; clarity gain is real (someone reading anthropicDefaults knows it's fallback).

### Commit F — Settings UI

Per-campaign provider + tier-model selection surfaced to the user.

**Deliverables:**
- `/campaigns/[id]/settings/page.tsx` — server component. Shows campaign's current provider + tier_models.
- Client component: provider dropdown (filters to `available: true` providers only — Anthropic at M1.5), three tier-model dropdowns (`fast`, `thinking`, `creative`) that repopulate when provider changes. Probe is not surfaced (technical-only).
- Save action: validates against registry; writes to `campaigns.settings`; invalidates turn cache if one's open.
- Soft warnings for incoherent combos (Haiku on thinking tier → "extended thinking is ignored by Haiku models").
- Link from `/campaigns` list to settings.
- Integration: switch a campaign's creative model between turns; next turn uses the new model.

**Audit focus:** dropdown repopulation doesn't leave stale selections; save flow validates server-side (not just client); unavailable providers visually degrade (disabled, not hidden — so users see what's coming); incoherent-combo warnings are soft, never hard-block.

## Acceptance ritual

1. Existing Bebop campaign migrated cleanly; plays unchanged.
2. Create a second campaign on Anthropic with Sonnet 4.6 for creative; play 5 turns; voice is audibly different from Opus Bebop.
3. Create a third campaign and attempt to pick Google as provider; gets helpful error about Google-KA landing M3.5.
4. Change Bebop's creative model from Opus 4.7 to Opus 4.5 snapshot mid-arc; next turn uses 4.5; voice shifts.
5. Langfuse shows `model` field on each agent span matches the campaign config.
6. Grep the codebase for `tiers.` — zero hits outside `anthropicDefaults.ts` / `providers/anthropic.ts`.
7. Write `docs/retros/M1.5.md`.

## What M2 / M2.5 / M3.5 use directly from M1.5

- **M2 (Session Zero):** SZ conductor reads the campaign's model context; prompts the user to confirm provider + model picks during SZ before handoff.
- **M2.5 (Billing):** cost metering aggregates by campaign → rolls up to user; the tier_models stay per-campaign so cost-per-turn varies with configuration. Billing substrate consumes the per-turn cost already-instrumented via Langfuse.
- **M3.5 (Google-KA):** populates the Google provider slot; flips `available: true`. Settings UI's Google dropdown comes to life without UI work.
- **M5.5 (OpenAI + OpenRouter):** same pattern.
