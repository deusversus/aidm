## What shipped

The walking skeleton's definition of done is satisfied. Every layer of the v4 stack is wired end-to-end, deployed, and exercised by at least one real request:

- **Infrastructure:** Next.js 15 + React 19 + TypeScript strict + Tailwind 4 + Biome + Drizzle + Postgres 16 + pgvector 0.8.2, all running on Railway's Dockerfile builder behind a public domain.
- **Auth:** Clerk v7 magic-link + Google OAuth; Svix-verified webhook upserts users into Postgres; protected route group behind `clerkMiddleware`.
- **Provider SDKs:** `@anthropic-ai/sdk`, `@google/genai`, `openai` wired as lazy singletons. `/api/ready` pings Haiku 4.5 with a 1-token call under a 3s timeout.
- **Observability:** Langfuse captured a hello-world Opus trace (`c5d8081d-...`). PostHog captures `$pageview` / `$identify` / autocapture / web vitals with Clerk-driven identity mirroring.
- **CI:** Biome lint + `tsc --noEmit` strict + Vitest smoke suite (27 tests, 1.3s) run on every push and PR. Node 22, pnpm-store cached via setup-node.
- **Typed contracts:** 24-axis DNAScales, 13-axis Composition, three-layer Profile/Campaign/ArcOverride model, hand-scored fixtures for Cowboy Bebop and Solo Leveling.

### What we built that wasn't in the original plan

The plan said "core Zod schemas." What we shipped in commit 7 was a **full re-architecture of v3's DNA and composition systems** — the product of a conversation that started with "why 11 DNA scales?" and ended with a cleaner three-layer model (canonical / active / arc_override) where DNA is explicitly prescriptive tonal treatment (not source description) and hybrid campaigns use `profile_refs: string[]` rather than any merge logic.

That re-architecture wasn't scope creep — it was recognizing that v3's DNA count and structure had landed through accretion, not theory. Doing it now means M1–M7 agents can consume the right contracts rather than propagating v3's legacy shape. Cost: ~2 extra hours of discussion. Benefit: every downstream agent gets a cleaner surface.

Specific outputs:
- 24 DNA axes across 7 groups (up from v3's 11), all orthogonal, all observable
- 13 composition axes (up from v3's 4), categorical enums for genuine discreteness
- Three-layer model: Profile's `canonical_*` fields (static), Campaign's `active_*` fields (persistent), ArcOverride (Director-writable, clears on transition_signal)
- Hybrid profiles via `profile_refs: string[]` — the Session Zero conductor (M2) authors the blend rather than any mechanical merge
- Delta helper (`dnaDelta`) for the Director to observe how far from source a campaign has drifted
- Hand-scored Cowboy Bebop and Solo Leveling fixtures that double as the M2 profile-research A/B eval ground truth

## What went well

**Memory discipline.** Memory files (`user_profile`, `v3_soul`, `audit_cadence`, `railway_next_build_env`) captured non-obvious knowledge as it surfaced. The Railway ARG gotcha in particular would have eaten another hour next time without the memory; now it's prescribed in the workflow.

**The subagent audit cadence.** Once established mid-M0, the pattern of `work → subagent audit → fix findings → push` caught three classes of bugs before they reached Railway: premature dep scope creep, missing `HOSTNAME=0.0.0.0` (which was the real cause of the healthcheck 404), and the `NEXT_PUBLIC_*` build-time inlining issue. Auditing via subagent — not the user — is the load-bearing discipline. The user's role is directional review of clean work, not defect hunting.

**Two-way conversation on architecture.** The user pushed back hard on my first two attempts to model DNA/composition layers. Each correction tightened the mental model. Getting the three-layer model right at M0 is categorically better than discovering at M4 that we need to add a layer.

**Provider SDK wiring as pure infrastructure.** Installing Anthropic, Google, and OpenAI clients as lazy singletons at M0 means M1's first real agent just imports and calls — no setup friction inside the feature work.

## What was harder than expected

**Railway + Dockerfile + Next.js standalone + non-root user** is a quartet of subtleties with compounding interactions. Six fix commits before prod served 200:

1. BuildKit cache mount rejected by Railway (`id=pnpm` needs a service-key prefix or removal) — removed
2. `output: "standalone"` forgotten in `next.config.ts` — added
3. Eager env parsing in `env.ts` crashed page-data collection (DATABASE_URL unset at build) — Proxy for lazy validation
4. Eager Pool construction in `db.ts` crashed same path (reads `env.DATABASE_URL` at module load) — converted to `getDb()` singleton
5. Missing `public/` directory broke final `COPY` in runner stage — added `.gitkeep`
6. `preDeployCommand: pnpm db:migrate` crashed runtime (drizzle-kit is a devDep, pruned; corepack can't download pnpm to nextjs user's home) — removed; migrations run from dev machine per DDD pattern

Then one more round after the first green deploy:

7. `HOSTNAME=0.0.0.0` missing → Next standalone bound to localhost → Railway proxy returned 404 on every request despite the container being "ready"
8. `NEXT_PUBLIC_*` vars never reached the build because Dockerfile didn't declare them as `ARG` — Next inlined `undefined`, PostHog silently no-op'd, Clerk's client fell back to server-only auth

Lessons:
- **DDD's Dockerfile is the reference.** Consulting it earlier would have saved most of #1–#6. The memory `project_railway_next_build_env.md` now prescribes ARG declarations for every new `NEXT_PUBLIC_*`.
- **The audit-before-push cadence works.** #7 and #8 were caught by a single subagent audit that ran on the full commit stack after I'd stopped iterating on deploy failures. The agent flagged HOSTNAME as the #1 likely cause of the 404 — correct on first try.

**Biome's `noDelete` rule vs. test patterns.** Tests that unset env keys via `delete process.env.X` triggered a Biome style error AND a TypeScript strict-mode error (NODE_ENV is augmented as non-optional by Next). Solution: `Reflect.deleteProperty(process.env, "X")` throughout. Small thing, but the bulk-replace cost a commit cycle.

**Scope creep pruning required judgment.** The subagent audit suggested removing ~10 deps from `package.json` as "premature." The ROADMAP explicitly said those deps land in M0 ("Provider SDKs installed and wired"). I went with the audit's recommendation (each dep lands in the commit that uses it) and flagged the judgment call to the user. Defensible either way. Future-me: when audit agent and ROADMAP disagree, surface the conflict to the user rather than picking silently.

## Lessons internalized (in order of impact)

1. **Audit via subagent BEFORE push, not after.** Running the audit on the whole commit stack after 5 deploy-failure commits would have caught HOSTNAME and ARG issues in the first round. The `feedback_audit_cadence.md` memory now prescribes audit as a mandatory pre-push step, not an optional post-hoc review.
2. **Check precedent projects (DDD, hvmsite) before reinventing deploy config.** The user told me this outright ("remember you can look at DDD"); the cost of forgetting was ~5 debug cycles.
3. **Railway + Dockerfile quartet:** (a) `ARG` every `NEXT_PUBLIC_*` in the builder stage, (b) `ENV HOSTNAME=0.0.0.0` in runner, (c) don't fight Railway's injected PORT, (d) no `preDeployCommand` pointing at devDeps. Memory captures this.
4. **Layer-collapsing is a conceptual trap.** I tried three times to fold DNA into composition (or vice versa) before understanding they're independent prescriptive dimensions. When the user's pushback sounds like "you're still not getting this," slow down and restate the layers before proposing merges.
5. **Thoroughness > minimalism for this user's audit cadence.** 24 DNA scales is not bloat if they're orthogonal and load-bearing. The user confirmed this directly: "103 would be bloat."

## Risks carried into M1

- **Mastra wiring is unproven.** M1 will be the first real Mastra workflow (the Intent → Outcome → KA cascade). The Claude Agent SDK + Mastra interop pattern isn't yet exercised. Spike before committing to a full implementation.
- **Prompt registry doesn't exist yet.** `lib/prompts/` is empty. M1 ships it alongside the first agents. The roadmap's Block 1 structure depends on having a fragment composition system in place.
- **First real Langfuse trace of an agent cascade (not a hello-world) will surface cost/latency reality.** Budget assumptions in ROADMAP §20 are untested.
- **Clerk's dev-instance behavior masks bot protection as 404.** When we try to automate E2E tests via curl against `pk_test_*`, the `dev-browser-missing` flow will bite. Use Playwright for any UI-driven test.
